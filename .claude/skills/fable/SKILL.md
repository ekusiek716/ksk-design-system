# fable — 進め方の規律(Phase 0→4)

このスキルを読んだ時点から Phase 0→4 の規律で作業を進めること。最初の行動は Phase 0 の3行を書くこと。

## 出力の規律(最初に読む)

書き出してよいのは次の3つだけ:
- (a) 作業記録(Phase 0 の3行・ピース一覧)。書き先は ①plan ファイルのパスが与えられていればそこ ②タスク管理ツール(Todo) ③どちらもなければ最終報告の冒頭に3行を再掲
- (b) 成果物そのもの(コード・ドキュメント・コミットメッセージ)
- (c) 最終報告(検証結果・未確認事項)

Phase 1 の仕分け・Phase 3 の自問(いずれも後述)は行動を選ぶための内部規律であり、その内容を応答本文に書き出さない。書き出すのは分岐の**結果**(止まる理由・ユーザーへの質問)だけ。

## 3原則

1. **完了条件を先に固定する。** 検証方法が具体的手段まで書けなければ理解不足とみなし、Phase 1 に戻って偵察する。偵察しても書けなければユーザーに聞く。
2. **自分の主張でなく観測を信じる。**
3. **次の一手は「いま最もリスクが高い場所」で選ぶ。**

用語(全 Phase 共通):
- **観測** = ツール実行の生の出力。テスト結果・エラーメッセージ・ファイルの実内容・grep のヒット・実際の画面。自分の記憶・推論・コードを読んだ印象・ドキュメントやコメントの記述は観測では**ない**。自分が直前に行った変更の効果も、観測するまでは未確認(「直したから直ったはず」は主張)。観測を省いて推論で代替した判断は、その時点で仮定に降格する。
- **リスク** = 間違っていたときの手戻りの大きさ × 観測でまだ確認できていない度合い。着手のしやすさ・所要時間の短さはリスクと無関係(順序の根拠にしない)。

例は全て同一のミニタスク「設定 `timeout` の変更が API 応答に反映されないバグの修正」から取る。完全版が必要なときだけ `references/worked-example.md` を読む(判定基準は本ファイルで完結)。

## Phase 0 — 触る前に3行を書く

コード・設定・データに触る前に、次の3行を書く(書き先は「出力の規律」の (a)):

- **完了条件**: 達成されるべき状態(What)。第三者が Yes/No で判定できる文か自己チェックする
- **検証方法**: その状態を自分で確認する具体的手段(How)。実行するコマンド・見る出力・操作する画面まで特定する
- **やらないこと**: 依頼に含まれそうで含まれない隣接領域を1つ以上(触らないファイル・変えない挙動・作らない抽象=今回はしない共通化・汎用化・将来対応)

依頼原文に明記された付随要望(「ついでに〜も」)は成果ごとに完了条件へ含め、Phase 2 で全成果のピースを1本のリスク順に並べる。明記のない推測の隣接領域は「やらないこと」へ。

- 良い例: 「完了: config の timeout を変えると /status の応答値が追随する / 検証: timeout=7 にして `curl :3000/status` が `"timeout":7` / やらない: 他の設定キーの読み込み方式は変えない」
- やってはいけない: 完了条件を「正しく動く」、検証方法を「動作確認する」と書く。手段が特定されていない=理解不足のサインであり、そのまま着手せず Phase 1 で偵察してから書き直す(偵察しても書けなければユーザーに聞く)。

**次へ**: 3行が書けたら Phase 1。

## Phase 1 — 偵察: 事実・仮定・不明に仕分ける

Phase 0 の完了条件に登場する名詞(ファイル・関数・API・データ)を、すべて**実物**で確認する。名詞が抽象的なら(例:「CSV 出力」)、それを実装している箇所を grep 等で特定し、実物のファイル・関数に置き換えてから確認する。実物=ソース本体・設定の実値・実行結果。コメント・型名・変数名・ドキュメントは誰かの主張であって実物ではない。修正タスクではまず不具合を実物で再現し、原因候補を仮定として挙げて観測で絞り込む(再現・特定できなければ「不明」として扱う)。

読みながら各命題を仕分ける(内部規律。仕分け表は出力しない):
- **事実** = いま裏を取った命題。裏取りに使った観測(コマンド・ファイル:行)を1つ指せるもの
- **仮定** = 検証可能だが未検証の命題。命名・コメント・記憶を根拠にした判断は全部ここ
- **不明** = 現時点で検証手段がない命題

最も危険なのは**事実のふりをした仮定**。見分け方: 「裏取りに使った観測を1つ言えるか」。言えなければ仮定。仮定は Phase 2 で検証されるピースに変換する。不明は、完了条件か検証方法の書き方を左右するものだけ即ユーザーに質問し(止まる条件の③)、左右しないものは最終報告の「未確認事項」に回して進む。

- 良い例: `loadConfig()` のコメントに「キャッシュなし」とあったが実装本体を読み、module スコープで一度だけ読んでいる行を確認してから事実に置いた。
- やってはいけない: コメントの「キャッシュなし」を信じて読み込み部を捜査対象から外す(コメントは主張。実装とズレていることがある)。

**次へ**: 名詞の実物確認と仕分けが終わったら Phase 2。

## Phase 2 — 分解: 独立して検証できる単位に、リスク順で

作業を「そのピース単独で、ツール結果1つで成功/失敗を判定できる」単位に割る。他ピースの完了を待たないと真偽が出ない割り方は独立ではない。順序は用語定義の**リスク**が高い順: 各ピースに手戻りの大きさ(大/中/小)と未確認度(高/中/低)を付け、両方高いものから並べる。同格なら依存の上流を先に。

- 良い例: 「A: 設定読み込み方式の変更(全 API に波及・未検証の仮定が集中)を先頭、B: エラーメッセージ文言の修正(影響が局所)を最後」
- やってはいけない: 5分で終わる B を「勢いをつけるため」に先にやる。easy-first は最大リスクを最後まで温存し、A が失敗した時点で B を含む作業全体が手戻りになる。

**次へ**: ピース列がリスク順に並んだら Phase 3。

## Phase 3 — 1ピースずつ: 実行→その場で検証

1ピースごとに「編集 → 最低1つのツール実行で結果を観測」する。観測するまで次のピースへ進まない。検証手段はそのピースの成功/失敗を判定できるものを選ぶ(挙動を変えるピースに grep の存在確認や lint 通過だけでは検証にならない)。**ピース完了の判定は記憶でなくツール結果と照合する**(「さっき直した」は判定に使えない)。

検証が赤なら: 赤のまま次のピースへ進まない。原因を Phase 1 の要領で観測し、必要ならピース列を組み替えて(Phase 2 相当)から再着手する。

各ピース完了ごとに、次の4つを自問してから次の一手を選び直す(内部規律。答えは出力せず、止まる条件に該当したときだけ理由を1行報告して止まる):
1. **観測は計画と矛盾していないか** — 矛盾したら Phase 2 の分解からやり直す。完了条件そのものが違ったと分かったら Phase 0 に戻る(=方針変更。止まる条件)
2. **いま最大のリスクはどこか** — 用語定義で再評価し、次のピースがそこでないなら並べ直す(並べ直したら先頭ピースから Phase 3 を継続)
3. **次の行動は可逆か** — 可逆=git で戻せる・データ破壊/デプロイ/外部送信を伴わない。不可逆なら止まる条件
4. **人間にしか答えられないことで詰まっていないか** — 該当例は「止まる条件」の③。推測で埋めずに止まる

- 良い例: 読み込み方式を直した直後に `curl :3000/status` で `"timeout":7` を観測してから、次のピースに進んだ。
- やってはいけない: 3ピースまとめて編集して一括テスト。赤が出たときどの変更が原因か切り分けられなくなる。

**次へ**: 全ピースが検証済み(緑)になったら Phase 4。未検証・赤が残る間は Phase 3 を出ない。

## Phase 4 — 全体検証と報告

1. **別レイヤーで確認**: 作った層の1つ外側から確認する(関数を変えたら呼び出し側から、CLI なら実コマンド実行で、API ならクライアント視点で)。Phase 3 で見た同じテストの緑を再掲するのは別レイヤー確認ではない(同じ根拠を二度数えない)。最外レイヤーがツールで観測できない(GUI での目視等)場合は、機械的に観測できる最も外側まで確認し、残る目視は「未確認事項」に書く。
2. **境界値で壊しに行く**: 空・0・負・最大・null/未設定・並行を試す。Phase 1 で仮定のまま未検証に残った項目を優先して壊す。
3. **報告の裏取り**: 報告に書く各因果主張(「X を直したから Y が直った」)に、それを示した観測を1つ紐付ける。紐付かない主張は「寄与は未確認」と書く。
4. **依頼原文との突き合わせ**: 依頼の言葉のまま全て満たしたか。Phase 0 の「やらないこと」に踏み込んでいないか。
5. **diff 通読**: 全 diff を読み、意図しない変更・デバッグ残骸・消し忘れがないか確認する。

- 良い例: ユニットテストの緑に加えてサーバーを起動し curl で観測(別レイヤー)。さらに timeout 未設定と 0 でも応答が壊れないことを確認した。
- やってはいけない: テストが緑だから完了とみなす。テストは自分が書いた仮定しか検証せず、別レイヤーと境界値が丸ごと抜ける。

1・2 で赤が出たら Phase 4 を完了とせず、Phase 3 の赤時と同じ要領で戻る。完了条件の解釈違いが原因なら Phase 0 へ(=方針変更、止まる条件)。

**次へ**: 5項目すべて済んだら最終報告を書いて完了。報告に含める: 完了条件ごとの検証結果(観測つき)/裏取りのない主張は「寄与は未確認」/境界値の結果/未確認事項/「やらないこと」に踏み込んでいない旨。

## 止まる条件(即停止してユーザーに確認)

- ① **不可逆操作**: git push・データ削除・マイグレーション実行・外部 API への書き込み・デプロイ・実ユーザーに届くメール/通知の送信(開発環境内で完結するテスト実行は除く)
- ② **方針変更**: Phase 0 の3行のいずれかを書き換えたくなったとき
- ③ **人間にしか出せない入力**: 要件の曖昧さの解消・優先度の判断・認証情報・本番データ

止まるときは「どこまで進んだか(観測つき)/何に詰まったか/選択肢」だけを短く報告する。
