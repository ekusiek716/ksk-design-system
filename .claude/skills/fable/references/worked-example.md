# worked example — 「設定 `timeout` の変更が API 応答に反映されない」を Phase 0→4 で通す

SKILL.md の各 Phase を1つのタスクで通した完全版。各所の表記:
- `[plan]` = plan ファイル / Todo に実際に書いた内容(出力の規律 (a))
- `[観測]` = ツール実行とその生の出力(要約)
- `(内部)` = 頭の中で行った判断。**応答本文には書き出していない**ことに注意
- `[報告]` = 最終報告に実際に書いた内容(出力の規律 (c))

依頼原文: 「config で timeout を変えても API のタイムアウトが変わらないみたいなので直してほしい。ついでにタイムアウト時のエラーメッセージが分かりにくいのも直したい」

---

## Phase 0 — 3行を書く

`[plan]`
- 完了条件: config の timeout を変更して API を再リクエストすると、応答の `"timeout"` と実際の打ち切り時間が追随する。タイムアウト時のエラーメッセージに上限値が含まれる
- 検証方法: `timeout: 7` に変更 → `curl :3000/status` が `"timeout":7` を返す。`timeout: 1` で遅いエンドポイントを叩き、1秒で打ち切られエラー文に「1s」が含まれる
- やらないこと: timeout 以外の設定キーの読み込み方式は変えない。リトライ機構は追加しない

(内部)「ついでに〜も」が2つ目の完了条件になるか迷ったが、依頼原文に明記されているので完了条件に含めた。リトライは依頼されていないので「やらないこと」に置いた。

## Phase 1 — 偵察

`[観測]` `grep -rn "timeout" src/` → `src/config.ts:12`(読み込み)、`src/api/client.ts:33`(使用)、`src/api/errors.ts:8`(エラーメッセージ)の3箇所。

`[観測]` `src/config.ts` を開く。コメントに「// 設定は毎回読み直す(キャッシュなし)」とあるが、実装は module スコープの `const config = readFileSync(...)` で**起動時に一度だけ**読んでいる(12行目)。

(内部)仕分け:
- 事実: 設定は module スコープで一度読み(config.ts:12 を確認済み)。エラーメッセージは errors.ts:8 の固定文字列
- 仮定→検証要: 「一度読みが原因で反映されない」はまだ仮定。client.ts が config を経由せず環境変数を直読みしている可能性も未排除
- 不明: 本番環境でプロセス再起動なしの設定反映が要件かどうか(依頼文からは読めない)

(内部)コメント「キャッシュなし」を信じていたら config.ts を捜査対象から外していた。実装本体を読んだから事実に置けた。

「不明」が完了条件の解釈に響くので、ここでユーザーに1問だけ確認: 「設定反映はプロセス再起動込みでOK? それとも稼働中に反映が必要?」→ 回答「再起動込みでOK。ただ再起動しても変わらないのが今のバグ」

`[観測]` 再起動しても変わらない、を自分でも再現: `timeout: 7` に変更 → 再起動 → `curl :3000/status` → `"timeout":30`。**再現した**。30 はどこから来るか grep → `client.ts:31` に `const DEFAULT_TIMEOUT = 30` があり、33行目が `config.timeout ?? DEFAULT_TIMEOUT` でなく `DEFAULT_TIMEOUT` を直接使っていた。原因はキャッシュではなく**設定値が未参照**。

(内部)Phase 0 の完了条件は変わらない(修正対象の見立てが変わっただけ)。方針変更には該当しない。

## Phase 2 — 分解

`[plan]` ピース(リスク順):
1. client.ts:33 を `config.timeout ?? DEFAULT_TIMEOUT` に変更(完了条件の本丸。全 API リクエストに波及・「これで直る」はまだ仮定)
2. タイムアウト実測の検証(timeout:1 で実際に1秒打ち切りされるか。応答フィールドと実挙動が別実装の可能性が未検証)
3. errors.ts:8 のメッセージに上限値を埋め込む(影響局所・依存なし)

(内部)ピース3が一番簡単で5分で終わるが、先にやらない。ピース1が失敗する(例: config の型が string で比較が壊れる等)場合、メッセージ側の修正方針ごと変わり得る。

## Phase 3 — 1ピースずつ

**ピース1**: client.ts:33 を修正。
`[観測]` `npm test` → 緑。`timeout: 7` で再起動 → `curl :3000/status` → `"timeout":7`。
(内部)4自問: 観測は計画と矛盾なし/最大リスクは次のピース2(実挙動が別実装かも)へ移った/次の行動は可逆(ローカル編集のみ)/人間待ちなし → ピース2へ。

**ピース2**: `timeout: 1` にして遅いエンドポイントを実測。
`[観測]` `time curl :3000/slow` → **30秒待たされた**。赤。応答フィールドは 1 なのに実際の打ち切りは 30秒のまま。
(内部)赤なので次に進まない。Phase 1 の要領で観測し直す: grep すると打ち切り実装は `client.ts` でなく `src/api/socket.ts:19` にもう1つあり、そちらも `DEFAULT_TIMEOUT` 直参照だった。「打ち切り実装は client.ts の1箇所」という仮定が崩れた → ピースを追加して仕分け直し。

`[plan]` ピース追加: 2a. socket.ts:19 も config 参照に修正(ピース2の検証はその後に再実行)

`[観測]` 2a 修正後 `time curl :3000/slow` → 1.0s で打ち切り。エラー文は旧文言。緑(打ち切りは完了条件どおり)。

**ピース3**: errors.ts:8 に上限値を埋め込み。
`[観測]` `npm test` → 緑。`curl :3000/slow` のエラー文に「timed out after 1s」。

## Phase 4 — 全体検証

1. 別レイヤー: ユニットテストの緑(Phase 3 と同じ根拠)は数えない。実サーバー起動+curl での実測(上記)がこれに当たる。さらに利用側スクリプト `scripts/healthcheck.sh` を実行し、変更後も exit 0 を確認
2. 境界値: `timeout: 0`(即時打ち切りになるか)/ timeout 未設定(DEFAULT_TIMEOUT=30 にフォールバックするか)を実測 → `[観測]` 0 は即時エラー、未設定は 30。仕様として妥当と判断、報告に記載
3. 報告の裏取り: 因果主張は2つ。「client.ts が設定未参照だったのが応答フィールド不追随の原因」← curl の前後比較で裏取り済み。「socket.ts の直参照が実挙動不追随の原因」← time curl の前後比較で裏取り済み。当初疑った「起動時一度読みキャッシュ」は再起動込み運用では**寄与は未確認**(修正もしていない)
4. 依頼原文: 「timeout が変わらない」→ 応答・実挙動とも追随を実測。「エラーメッセージ」→ 上限値入りを確認。「やらないこと」(他キーの読み込み方式・リトライ)には触れていないことを diff で確認
5. diff 通読: `git diff` 全読 → client.ts / socket.ts / errors.ts の3ファイルのみ。デバッグ用の console.log が socket.ts に残っていたのを発見・削除して再テスト緑

`[報告]`(最終報告の骨子)
- 原因は2箇所の `DEFAULT_TIMEOUT` 直参照(client.ts:33 / socket.ts:19)。config 参照に修正し、応答フィールドと実打ち切りの両方が追随することを curl / time curl で実測
- エラーメッセージに上限値を追加
- 境界値: timeout:0 は即時打ち切り、未設定は 30s フォールバック(実測)
- 未確認事項: 稼働中(無再起動)の設定反映は対象外(ユーザー確認済み)。当初疑った起動時キャッシュの寄与は未確認

---

## この例が示していること

- コメント(「キャッシュなし」)と依頼文の推測(「キャッシュが原因?」)はどちらも主張であり、観測(再現・grep・前後比較)が原因を2回訂正した
- easy-first を避けたことで、ピース3の手戻りがゼロで済んだ
- 赤(30秒待ち)で立ち止まって仕分け直したから、socket.ts の2箇所目が見つかった
- 仕分け・自問は plan と報告に**結果だけ**が現れ、思考の実況は出力されていない
